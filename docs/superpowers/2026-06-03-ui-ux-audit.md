# UI / UX audit — Munbeop Garden v3

> Generated by `impeccable` (critique + audit pass) on 2026-06-03 against `https://mungarden.vercel.app/` (commit `2ab74c4`) and the local source tree at `munbeop/`.
> Register: **product** (app UI serves a task; not a brand landing).
> Surfaces inspected: `/`, `/practice`, `/library`, `/log`, `/stats`, `/settings`, `/auth/sign-in` × light + dark × desktop (1280) + mobile (390).
> **Two passes:** initial audit (sections below up to the plan), plus a Pro Max addendum at the bottom with measured contrast ratios, keyboard nav probes, security findings, and motion/i18n issues caught from interactive testing.

## TL;DR

**Audit Health Score: 13 / 20 — Acceptable** (significant work needed before users are happy).
**Design Health Score (Nielsen 10 × 4): 23 / 40 — Acceptable** (solid foundation, several heuristics under-built).
**Anti-pattern verdict:** does NOT look AI-generated. The LADX-anchored pixel-art aesthetic is a real, committed brand choice and avoids the 2026 saturated AI cream-default trap by being specific. The implementation, however, partially undoes that work by bypassing its own token aliases, shipping bright-jade buttons that fail WCAG AA, and leaving primitive coverage at 6 of 12 specced.

The design system spec (`docs/superpowers/specs/2026-06-03-design-system-v2/*`) is world-class. The implementation lags behind it by one polish pass.

## What's already excellent

1. **The design system spec.** Five files (overview, tokens, primitives, chromes, sprites + mascota) totalling >2 000 lines of considered design. The token layer cleanly separates brand swatches from semantic aliases, WCAG ratios are documented per pair, the dark theme is a curated "night palette" that re-points aliases instead of doing a cheap CSS invert, and the migration table from v1 → v2 is column-by-column. This is the kind of artifact teams ship for years without ever writing down.
2. **Register choice is brave and correct.** LADX as the anchor is a specific anti-reference, not "cozy / warm / handcrafted" hand-wave. The cream isn't an AI default — it's the actual `#f8efd0` parchment from the GBC cart. Pixel art + Press Start 2P + Silkscreen + Noto Sans KR is a committed brand identity, not a safe-bet stack.
3. **Bilingual KO + Latin title pattern works.** Every page pairs Hangul (Noto Sans KR 900) with the Latin word in Press Start 2P at a subordinated size. It's the brand's actual differentiator vs Duolingo / Anki, and the typography hierarchy reads correctly across all eight locales.
4. **Dark theme is genuinely dark.** Not Discord gray. The cave-brown `#1a1612` with bright accents inheriting from light theme is recognizable as a sibling of the light theme rather than an inversion of it.
5. **SRS algorithm + practice 3×3 loop is the right cognitive size.** Three grammars × three contexts = nine sentences fits a single sitting. The easy / hard + post-review note + error block is a pedagogical pattern Duolingo doesn't have.

## Priority issues

### P0 — block before next ship

#### [P0] Primary button fails WCAG AA contrast
- **Location:** [Button.vue:30-45](munbeop/app/components/ui/Button.vue:30)
- **Category:** Accessibility (also Theming)
- **Impact:** The most-used CTA in the product (Sign in, GO, EASY, Submit, Restart, Save with note) renders cream-on-bright-jade at ~2.9:1 — fails both AA (4.5:1) and AAA (7:1). Sam screen-reader / low-vision users miss the primary action. The implementation chose `background: var(--jade)` (#3aa84a) + `color: var(--paper)` (#f8efd0), bypassing the semantic aliases `--accent` (jade-deep #185f24 = 6.2:1 PASS) and `--text-on-accent` (paper). The token spec `01-tokens.md §1.4` documents this exact combo as a FAIL — the implementation contradicts its own contract.
- **Recommendation:** Rewrite `Button.vue` to consume `--accent` / `--text-on-accent` / `--danger` / `--text-on-danger`. Same change kills the symmetric problem on `danger` variant. Add `:focus-visible` outline (currently missing). Add `size` / `loading` / `fullWidth` props per spec `02-primitives.md §1`.
- **Suggested command:** `/impeccable polish app/components/ui/Button.vue` (or `/impeccable harden` for the focus + loading additions)

#### [P0] Local `pnpm dev` is broken on the worktree
- **Location:** root + `nuxt.config.ts`
- **Category:** Performance / DX (blocker for verification work)
- **Impact:** Every route returns 500 "Package import specifier '#vite-node' is not defined" — a known Nuxt 4.4.6 + pnpm `imports`-map regression when pnpm hoists `vite-node` away from `@nuxt/vite-builder`. Blocks any in-browser audit, screenshot, or verification work locally. Forces this very audit to run against the prod URL.
- **Recommendation:** Pin `nuxt` to a version with the fix (4.5.x line), OR add a `.npmrc` `public-hoist-pattern[]=vite-node`, OR upgrade to Nuxt 4.6+. Add CI step that runs `pnpm dev --port 3001` headless and curls `/` for a non-500 response so this regression is caught.
- **Suggested command:** out-of-band fix; no impeccable command applies.

### P1 — major issues

#### [P1] Sky used as text colour fails contrast
- **Locations:** [ContextDisplay.vue:42-46](munbeop/app/components/practice/ContextDisplay.vue:42), [sign-in.vue:112-115](munbeop/app/pages/auth/sign-in.vue:112), [AccountWidget.vue:67](munbeop/app/components/layout/AccountWidget.vue:67)
- **Category:** Accessibility (token contract violation)
- **Impact:** `color: var(--sky)` on `--paper-warm` ≈ 2.0–2.3:1 (FAIL). The token spec `01-tokens.md §1.3` rule 1 says verbatim: *"Text only uses `--text`, `--text-soft`, or a `--text-on-*` alias. Never a brand swatch directly, never `--jade`/`--red`/`--gold`/`--sky` as text."* Three real components break this.
- **Recommendation:** Replace with `--ink` (body) or `--text-soft` (helper). Use the sky colour as a 1 px left rail or a small icon, not as type.
- **Suggested command:** `/impeccable polish app/components/practice/ContextDisplay.vue app/pages/auth/sign-in.vue app/components/layout/AccountWidget.vue`

#### [P1] Card shadow is too faint to carry the LADX silhouette
- **Location:** [Card.vue:47](munbeop/app/components/ui/Card.vue:47)
- **Category:** Theming (system token bypass)
- **Impact:** `box-shadow: 4px 4px 0 var(--paper-deep)` paints a cream-on-cream block. The shadow that's supposed to make every card sit on the parchment with a chunky pixel anchor is visually missing. Affects every page (sign-in card, library grid, practice cards, completion banner, settings rows).
- **Recommendation:** Replace with `var(--shadow-card)` (which resolves through `--shadow-color` = `--ink-soft` in light, near-black in dark — both theme-correct). Same change for the `card--hover` state.
- **Suggested command:** `/impeccable polish app/components/ui/Card.vue`

#### [P1] Game-chrome corner ornaments invisible in dark mode
- **Location:** [Card.vue:17-35](munbeop/app/components/ui/Card.vue:17)
- **Category:** Theming
- **Impact:** All four corner ornaments hardcode `fill="#185f24"` (jade-deep). In dark mode, `#185f24` on `#1a1612` cave bg ≈ 2.6:1 — the ornaments effectively vanish. Game-chrome is supposed to scream "you're in a Zelda dialog now"; in dark mode it whispers.
- **Recommendation:** Switch `fill` to `currentColor` and set the wrapper's `color: var(--accent)` (light = jade-deep, dark re-points to jade #3aa84a per `colors-dark.css`). Ornaments become theme-aware automatically.
- **Suggested command:** `/impeccable polish app/components/ui/Card.vue`

#### [P1] Mobile navbar drops Journal / Log and ships icon-only
- **Location:** [MobileNavbar.vue:10-15](munbeop/app/components/layout/MobileNavbar.vue:10)
- **Category:** Responsive / Recognition vs Recall
- **Impact:** Five items on mobile vs six on sidebar (Log missing). Users who practiced on mobile can't find their journal. Jordan first-timer also has no labels — five glyphs (cottage / die / books / bar chart / cog) and they're 22 px crispEdges pixel sprites that don't all telegraph their meaning at that size.
- **Recommendation:** Add the Log entry back. Either add inline 9 px Press Start 2P labels under each icon (matches the brand), or surface the labels in long-press tooltips for thumb users. Reserve at least 56 px navbar height (currently 60 px — fits).
- **Suggested command:** `/impeccable adapt app/components/layout/MobileNavbar.vue`

#### [P1] LocaleSwitcher renders the native browser `<select>`
- **Location:** [LocaleSwitcher.vue:42-52](munbeop/app/components/layout/LocaleSwitcher.vue:42)
- **Category:** Consistency and Standards
- **Impact:** Native dropdown sits inside an otherwise-committed pixel-art surface — looks like an OS chrome element pasted on a parchment scroll. Appears twice (sidebar bottom + Settings card). Dark mode makes it worse (white control on dark bg).
- **Recommendation:** Build a `Tabs`-like pixel-art combobox, OR (cheaper) style the native `<select>` with `appearance: none` + pixel border + pixel chevron sprite. Per spec there's no `Select` primitive — add one to `02-primitives.md` if you go the proper route.
- **Suggested command:** `/impeccable polish app/components/layout/LocaleSwitcher.vue` (style native) or `/impeccable shape <Select primitive>` (build new)

#### [P1] Practice session shows 9 sentences at once with no pagination
- **Location:** [practice.vue:52-65](munbeop/app/pages/practice.vue:52)
- **Category:** Cognitive Load / Aesthetic and Minimalist Design
- **Impact:** Three `GrammarCard` × three contexts inline = nine cards stacked on one scroll. Desktop is overwhelming; mobile is a thumb marathon. Each card has Korean grammar + meaning + example + 3 progress dots + context + textarea + Easy/Hard buttons. Casey can't keep state in working memory between cards.
- **Recommendation:** Sequential single-card view with a top progress bar (`current / 9`). Mascota peek inside the active card's `#peek` slot per `03-chromes.md`. Add keyboard shortcuts (`Enter` = Easy, `Shift+Enter` = Hard, `Tab` between contexts) for Alex power-users.
- **Suggested command:** `/impeccable shape practice-session-redesign` then `/impeccable craft`

#### [P1] Pages bypass primitives systematically
- **Locations:** every page in [pages/](munbeop/app/pages/) re-implements `.title` (BilingualTitle missing), `.card` / `.empty` div (Card primitive missing), `.label` (no Field primitive)
- **Category:** Consistency and Standards
- **Impact:** Six pages duplicate the same ~30 lines of bilingual title CSS. Five pages reinvent the empty-state card box. Auth page hand-rolls form labels. Adoption of the design system is partial; every page has a slightly different idea of "the empty card." Token migrations (theme, palette) won't cascade because the consumers aren't going through the system.

### P2 — minor

#### [P2] ProgressDots uses banned elastic easing
- **Location:** [ProgressDots.vue:31](munbeop/app/components/practice/ProgressDots.vue:31)
- **Impact:** `cubic-bezier(0.3, 1.3, 0.5, 1)` — the `1.3` overshoot is the impeccable absolute-ban "No bounce, no elastic." Also `transition: all` is the lazy-write.
- **Recommendation:** `transition: transform var(--motion-quick) var(--ease-out), background var(--motion-quick) var(--ease-out);`. Drop the overshoot.

#### [P2] ProgressDots "current" dot uses --red
- **Location:** [ProgressDots.vue:34-37](munbeop/app/components/practice/ProgressDots.vue:34)
- **Impact:** Red semantically = danger / hard feedback (used by the HARD button). Using it for "currently active" overloads the colour. The done dots use `--jade`, so the active dot expecting `--accent` (jade-deep) is the obvious system pick. Also `box-shadow: 0 0 0 4px rgba(194,62,62,0.2)` uses the v1 red RGB — migration leak.
- **Recommendation:** `--accent` for current; halo via `--accent` at low alpha or removed entirely (the scale(1.3) already signals).

#### [P2] v1 RGBA leaked into v2 components
- **Locations:** [ContextDisplay.vue:26](munbeop/app/components/practice/ContextDisplay.vue:26), [ErrorNoteBlock.vue:38](munbeop/app/components/practice/ErrorNoteBlock.vue:38), [ProgressDots.vue:36](munbeop/app/components/practice/ProgressDots.vue:36)

- **Recommendation:** Add a token for tinted-accent backgrounds (e.g. `--surface-info-soft: color-mix(in oklch, var(--sky) 8%, var(--paper-warm))`) and consume that instead of hand-mixed RGBA.

- **Location:** [Input.vue:40](munbeop/app/components/ui/Input.vue:40)
- **Impact:** `font-size: 15px` is neither `--text-base` (14) nor `--text-md` (16). Spec `01-tokens.md §3.2` explicitly enumerates the eight steps.
- **Recommendation:** Use `--text-base` for compact inputs, `--text-md` for the practice textarea (long reading).

#### [P2] Mascota brim shadow inverts in dark mode
- **Location:** [Mascota.vue:55](munbeop/app/components/practice/Mascota.vue:55)
- **Impact:** Hat brim shadow hardcodes `#4a3a1f` which equals `--ink-soft`. The brand swatch inverts to dusty cream (`#d8c89a`) in dark, but the SVG hex is frozen. Mascota looks slightly off in dark — the under-brim shadow is now darker than it should be relative to the cave-brown bg.
- **Recommendation:** Wrap the SVG group in `<g fill="currentColor">` with `color: var(--ink-soft)` on the host, or duplicate the path under `[data-theme="dark"]` with the theme-correct hex.

#### [P2] Toast missing variants, stacking, progress, aria-live
- **Location:** [Toast.vue](munbeop/app/components/ui/Toast.vue)

Implementation is single-message single-variant single-instance. Sign-in error path uses it for genuine errors but loses the assertive announcement.

## Patterns and systemic issues

1. **Token alias layer is bypassed.** The spec's leverage comes from components referencing `--accent` / `--text-on-accent` / `--surface` / `--shadow-card` so the token layer absorbs theme switches. The implementation references raw brand swatches (`--jade` / `--paper` / `--ink`) in `Button`, `Card`'s shadow, `Input`'s bg, `Sidebar` brand, `Toast` bg, every page. The dark theme works mostly by accident; one swap that the spec routes through, the components don't. **Net effect:** every component is a dark-mode bet.
2. **6 of 12 primitives shipped.** Missing: `Field`, `Modal`, `Tabs`, `Badge`, `Avatar`, `Tooltip`, `BilingualTitle`. Each absence is felt as inline re-implementation (titles in 6 pages, empty-state cards in 5 pages, form labels in `sign-in.vue`). Build the next batch and most page churn goes away.
3. **Specs > implementation.** Plan 3 produced a world-class design system; the implementation pass is one round behind. The fix is mechanical, not creative — the spec already says what to write.
4. **Empty surfaces dominate.** Garden, Stats, Settings, Log (when empty), Practice (intro) are 70 %+ whitespace on desktop. No second-tier content (recent activity, contextual help, "next up"). The app feels less complete than the legacy v2.22 — which was a god file but at least filled the screen with stats / collection / log.
5. **No onboarding / first-run.** First-time user lands on Garden with the placeholder greeter, has to read the copy, find the `→ Practice` link, click GO, then face nine sentences. No tour, no "what is this?", no example sentence demonstrating the loop. Jordan abandons before reaching the first grammar card.

**Casey (distracted mobile user):**
- Mobile sidebar swap works, but the 60 px bottom navbar is fine for thumb. Active state is a 3 px top border — easy to miss in glance.
- Practice session = nine cards stacked vertically on a 390 px viewport. Easy to lose your place between interruptions; no auto-save on the input field.
- Sign-in's two-input + button block sits 250 px from the top — thumb-reach OK on a 5.5" phone, harder on a 6.7" iPhone Pro Max.

## Improvement plan (proposed order, with `/impeccable` commands)

Each phase is a candidate for its own PR. Phases 1–2 are blockers; 3–5 are the big lift; 6–8 are polish.

### Phase 1 — Unblock and pass contrast (PRs ~ small)

## B. Keyboard navigation walkthrough

Computed by enumerating all `[tabindex]`, `a[href]`, `button`, `input`, `textarea`, `select` in DOM order on `/practice` after GO, light theme, desktop.

### Tab stops (16 visible interactives in the session view)

1–6. Sidebar nav (6 items) → 7. AccountWidget Sign in → 8. LocaleSwitcher select → 9. Card 1 textarea → 10. Card 1 EASY → 11. Card 1 HARD → 12. Card 2 textarea → 13. Card 2 EASY → 14. Card 2 HARD → 15. Card 3 textarea → 16. Card 3 EASY → 17. Card 3 HARD.

### Findings

- **No `:focus-visible` rule on `Button` at all.** Computed CSS rules show only `.button { ... }`, `.button:hover:not(:disabled)`, `.button:active:not(:disabled)`, `.button--secondary`, `.button--danger`, `.button--disabled`. Keyboard users see the same visual as non-focused; the browser's default focus ring is suppressed by the `outline: none` implicit on a custom-styled button. **Verdict:** keyboard-only users cannot tell where the focus is on any button in the app.
- **No `:focus-visible` rule on `Card` either.** Card never reacts to hover or focus. Library is a grid of 14 cards; a keyboard user has no visible indicator while tabbing through.
- **No `:focus-visible` rule on `.sidebar__link`.** Sidebar items have a `:hover` rule (paper-deep bg + ink color) but Tab focus is silent.
- **No keyboard shortcut for submit.** `Enter` in the practice textarea inserts a newline (correct multiline default). `Shift+Enter` does nothing. No `1` / `2` / `e` / `h`. Alex power-user has no path.
- **No focus trap, no `Esc` handler** — but there are no modals yet, so this is moot until `Modal` ships.
- **Tab order is correct visually** (top-to-bottom, left-to-right) but the LocaleSwitcher (#8) is in the sidebar footer — a long tab-jump from "Sign in" to "Card 1 textarea" via a control most users never touch mid-session. Minor friction; not a blocker.

## C. Practice loop in motion

Executed: GO → type `저는 학생입니다.` → click EASY → observe → type `저는 한국어를 공부해요.` → click HARD → observe error note block.

### EASY submit findings

- Card 1's textarea **cleared instantly** (no transition, no acknowledgement).
- Progress dots advanced: `[done, current, empty]` for Card 1 (one of three contexts done).
- Mastery badge stayed `Sprout · 새싹` (no animation, no peek of "+1 toward Plant"). Per spec the threshold is ≥20 reviews + easy ≥ hard × 1.5, so no level change is correct; but there's no signal that the user moved *along* the path. Germane cognitive load — the *learning* effort — is unrewarded visually.
- **No toast visible at screenshot time** (~250 ms after click). The `useToast` composable is called with `t('practice.toast_saved_easy')`; the Toast component renders a Vue `<Transition>`. Either the toast appeared and faded faster than 250 ms (Transition duration is 300 ms total), or `useToast` short-circuits. Either way, the visual confirmation channel is too weak — a beginner doesn't know if their submission registered.
- The mascota **stayed in the same idle pose** above the cards. Spec `04-sprites-mascota.md §5` envisages emotional expressions per outcome; the placeholder skips this entirely, but even the placeholder could bob / blink on save.

### HARD submit findings

- The `ErrorNoteBlock` **opened *below* the EASY/HARD button row**, pushing the card height by ~150 px. On mobile that pushes Card 2 off-screen further; on desktop it works.
- The textarea above keeps the user's typed sentence — good (Casey persona: state preserved on interruption).
- Two CTAs: "Save with note" (red `danger`) + "Skip" (cream `secondary`). Red is the visually loudest option, which means "Save with note" reads as "destructive". Semantically it's the recommended path (user took time to write a note about their confusion). Swap: "Save with note" should be `primary` (jade) and "Skip" should be `secondary`. Currently the colour vocabulary reverses pedagogical intent.
- No character count / max length on the note textarea — Riley stress-tester can paste an essay.

### Failed sign-in path

- Two `<input>` elements on `/auth/sign-in` are **both `type="text"`** — the password field is **plaintext** as the user types. Browser password manager will not offer autofill. Shoulder-surfers can read the password. Mobile keyboard does not switch to password-safe layout. **Security + UX P0.**
- Submitting with `fake@test.com` + `wrongpw`: the button briefly disables, the request goes to Supabase, returns an error… and within 2 seconds **no toast surfaces** in the DOM. Either the toast came and went inside the Transition window, or the error handler doesn't fire. Either way, the user sees nothing change. There is also no inline form error — the inputs keep their bad values with no annotation.
- The form has no `<form>` element — it's three `<button>`s manually wired. `Enter` inside the email/password input does nothing; users have to mouse-click "Sign in". Sam persona red flag for keyboard-only login.

Computed by scraping the parsed CSS rules attached to `.button`, `.card`, `.sidebar__link` selectors.

| Primitive | rest | hover | focus-visible | active | disabled |
|---|---|---|---|---|---|
| Button | bg `var(--jade)`, color `var(--paper)`, shadow `4px 4px 0 var(--ink)`, transition `transform 0.08s, background 0.15s` | translate(-1,-1), shadow 5px | **missing** | translate(2,2), shadow 2px | opacity 0.4 |

### Specific issues uncovered

- **`Button` shadow uses `var(--ink)`** = `#1a1a1a` (pure ink) in light, `#f8efd0` (cream) in dark. Light mode pure-black shadow is loud and breaks the warm LADX palette (spec calls for `--shadow-color` = ink-soft `#4a3a1f` = warm brown). Dark mode would render a *cream* shadow against cave-brown bg — that's a halo, not a chunky-pixel anchor. The spec's `--shadow-color` exists precisely to fix this; the implementation bypasses it.
- **`Button` transition durations are hardcoded** (`0.08s, 0.15s`) instead of `var(--motion-quick)` / `var(--motion-base)`. The motion token system has no effect on the primary CTA.
- **`Card` has no hover behaviour at all.** A clickable card (e.g. Library grid) gives no visual feedback under the mouse. Hover affordance is a basic-product expectation.

## E. Internationalisation pressure test

Switched the UI locale to **Japanese** and **Thai** via `LocaleSwitcher`.

- All UI strings translated correctly (nav, button labels, copy, sidebar widget) — `@nuxtjs/i18n` is wired correctly.
- **Press Start 2P font has no CJK glyphs.** When the locale is `ja`, the nav items "練習" "図書館" "統計" "日記" "設定" render in the **fallback font** (`monospace` declared after PS2P), which on Windows resolves to MS Gothic / on macOS to Hiragino. Result: the pixel-art nav labels look smooth / anti-aliased on JA. Same for `th` (PS2P has no Thai glyphs either — Thai text falls to system Sans). The pixel-art aesthetic collapses on five of the eight locales (`th`, `ja`, `vi`, `id`, `pt-BR` partial accents). EN/ES/FR are the only fully pixel-rendered locales.
- **"화이팅! 💪"** literal Korean string is preserved verbatim in the Japanese translation of `practice.intro_lead`. Cultural retention is intentional in some cases ("Hwaiting" as a Korean cheer); for a Japanese user this means an unexplained foreign word inside their UI. Either translate ("頑張ろう!") or annotate (ruby `<rb>화이팅</rb><rt>hwaiting</rt>`).
- **No Korean as UI locale** — the eight locales are all *consumer* languages, not Korean. This is intentional (Korean is the target language, not the chrome) and correct. Worth noting in `PRODUCT.md` so future contributors don't add it.
